# 工作总结

当用户要求汇报当天工作、当前会话、阶段进展、实现结果或后续计划时，使用此技能。

## 汇报原则

- 结论先行：第一句话直接说明当前结果、完成度或核心状态。
- 简洁聚焦：只写对决策、验收、推进有价值的信息，删掉过程闲话。
- 抓重点：优先写完成的能力、关键逻辑、数据或状态变化、验证结果、风险。
- 抓进度：明确哪些已完成、哪些进行中、哪些待处理。
- 不编造：没有证据的工作不要写；缺少记录时直接说明无法确认。

## 生成步骤

1. 从对话、工具动作、文件修改、测试结果和决策中提取实际完成事项。
2. 合并同类工作，按“结论、进度、验证、风险、下一步”筛选内容。
3. 用业务或功能语言描述变化：系统现在能做什么、流程如何推进、结果如何体现。
4. 只在影响理解或验收时说明技术实现细节。
5. 输出前压缩措辞，去掉代码细节、命令回显和无关背景。

## 输出格式

默认使用中文。除非用户另有要求，只输出两个部分：

```markdown
工作总结
...

明日计划
...
```

- `工作总结`：约 100-150 字，使用 3-5 条短句或一个紧凑段落。
- `明日计划`：50 字以内，只写下一步可验证动作。
- 工作总结应包含结论和进度；验证、风险只在确有发生或仍需关注时写。
- 可使用“已完成 / 进行中 / 待处理”表达进度，但不要为了格式牺牲简洁。

## 禁止内容

- 不写代码名、文件名、分支名、包名、函数名、类名、变量名、提交号、工单号或内部项目代号。
- 不暴露原始命令输出，除非用户明确要求。
- 不写空泛表述，例如“持续优化”“推进相关工作”，除非后面有具体对象和结果。
- 不把计划写成愿望，要写成可检查的动作。
